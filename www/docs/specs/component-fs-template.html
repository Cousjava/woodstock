2.0 Requirements, goals, and assumptions

2.1 Requirements

Include name of requesting group/person, description of requirement.  State each requirement clearly.  If there is a more detailed description of the requirement in another document, you may paraphrase it here, or use a link (with anchor if possible, or a requirement number otherwise) to the requirement in the other document.  This is especially useful for user interface requirements listed in the UI design document.

All requirements should be numbered.  Each requirement should be clear and succinct, and should be stated in the form or a requirement (e.g., use MUST, SHOULD, MAY, etc.).

2.1.1 UI requirements

Note that the component developer is expected to work with the an appropriate UI engineer to make sure requirements for good user interface design are included.  

Links to specific sections of the UI guidelines, UI design document, and/or screen shots would go in this section.

Some sections may be omitted.  If this specification is for changes to an existing component, it is not necessary to repeat the requirements for that component here; just those requirements related to the changes.

List explicit functionality this component must, should, or may support.

2.1.3 Performance requirements

List any special requirements for performance for this component.

  • The table of supported Browsers.

Document any exceptions to the standard Woodstock supported browsers and platforms.

2.1.5 Special requirements

Include any special requirements for A11Y, L10N, online help, etc.  We assume all components will meet the basic requirements for accessibility, localization, XHTML compliance, etc.

2.2 Requirements Not Met

Include requirements that will not be met by this version of the component.  Include requesting group/person and description.  If possible, include a brief description of why the requirement can not be met.

2.3 Assumptions

Include any assumptions you are making that are not explicitly stated as requirements or goals.  These are not explicit dependencies on other software, components, or infrastructure; they will be covered in the Dependencies section.

2.4 Goals

Include any specific goals not covered by requirements.  Goals may include features, design approaches (like Ajax), etc. for which you wish to call explicit attention.  They may explain motivations for why some functionality is included.

2.5 Non-goals

Include specific non-goals; basically to draw attention to some point that is important to the features, design, or approach you are taking.  They may explain motivations for why some functionality is not included.

3.0 Component Usage

Include descriptions of how the component is expected to be used in web applications.  Include some example JSP code or Javascript code (if useful).   This section is important to provide a general understanding of the component and how it is meeting the requirements.  It is expected to provide input to the QA engineer developing the test specification and the technical writer developing user documentation.

Reference UI specification behaviors and interaction patterns (links can be used).

Include how the component may be used in conjunction with other components; e.g., when used in a table or other complex component. 

If the component will use Ajax features, include descriptions of Ajax will be used to provide functional features of the component.  Identify data and control flows where useful to explain how a component works.

If the component will define and use facets, explain the expected behavior of components included in those facets.  What is the "contract" to be fulfilled by the components in the facet?  Give some examples.

Any there any special configuration steps for this component - a filter, backing bean definition, provider config, etc?

4.3 Component interfaces

Define all public component classes, methods, and inheritance.  Full functional descriptions are required, including a description of the semantics of each class and function.  Public means those interfaces exposed to web application developers and other components; not necessarily Java class methods declared as public.  If a Java method must be public, but should not be exposed to web application developers, it may need to be moved to a separate class.

Details of the interfaces declared in section 4.1 should be included here.

If a link to a javadoc is specified, the javadoc must meet the requirement for full descriptions.

(Note: you can NOT use a link to a javadoc in the build environment!  The javadoc must be static and revisioned along with this specification.)
