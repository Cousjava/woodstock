  1.0 Overview

          This test specification describes the approach to be taken when testing the rendering of the 4.2 Project Woodstock Login Component.

1.1 Component Description

This document specifies the Login user interface component. There are two sets of interfaces in this component. A visual interface that both the user and  the web application developer sees and a back end interface that application developers and authentication service providers need to be aware of.

This component is heavily dependent on the Java Authentication and Authorization Service (JAAS) - specifically the authentication part of JAAS. The login presentation layer and a corresponding JAAS based authentication service is currently part of the webconsole's infrastructure. Internally within Sun a lot of the web applications are not able to or are unwilling to be part of this console and, hence, take advantage of its authentication service. These applications still need users to authenticate before being allowed to run the application. Other web applications may need an authentication presentation mechanism for primary or secondary authentication or even a portal channel for authentication. Most if not all of these situations can be addressed by an independent login presentation component. This component up levels the pluggable authentication mechanism to the web layer.

Ajax is being used in this component in order to allow the authentication infrastructure to drive the presentation - this was not possible before because of the nature of the web environment - a HTTP get had to be done before any meaningful HTTP post could occur which meant the first login page to be displayed was not being driven by the auth infrastructure. Various tweaks including dummy presentation layers were put in place to get around these problems earlier.

1.2 Component Purpose

There were two main purposes of this component:

a) To allow the actual authentication service to drive the presentation.

b) To have a component that is not tied to any presentation framework. So far every authentication technology that was part of the login pages had its own presentation layer. This component obviates the need for such a page.

2.0 Test Requirements

The Login component will be tested to meet the following requirements, standards, and tests:
  • SWAED Web App Guidelines compliant
  • Fully A11Y compliant by GA
  • I18N compliant by GA
  • XHTML transitional compliant by GA
  • GUI functional test requirements
  • JSP test requirements
  • API test requirements

2.1 Tools and Methods:

The Login component will be tested using the the following tools and methods:
  • GUI tests will be performed on platform and browser configurations identified here:
    • The table of supported Browsers.
    • The table of supported Containers.
  • Functional and BUI tests will be automated
  • API tests will be written for each method in the classes of the component
  • Automated tests will be integrated into a test harness and run nightly as a regression suite
  • Automated code coverage goal of 80% (at GA) will be achieved as outlined in the project test plan

3.4 API Tests

API level unit tests will be written by QA, calling each method written for the component.  These tests include both positive and negative value arguments.  The goal is to look for consistent and proper behavior, making sure range limits are met, and correct error reporting is handed by the class. The API tests will be incorporated into the internal test harness and run nightly.   Nightly test results will be posted and available for review.  JCOV, the java code coverage tool, will be used to ensure that we meet the code coverage goales for milestones and GA in particular.
