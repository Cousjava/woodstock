1.0 Overview

This test specification describes the approach to be taken when testing the rendering of the 4.3 Project Woodstock Rating Component (Rating).

1.1 Component Description

The Rating component displays a series of images, each of which signifies a rating (or grade) level.  There are two sets of interfaces in this component:  A client-side javascript interface that allows a user to dynamically manipulate the widget, and a server-side JSF based component interface.    This component will be made available in the Woodstock component library and in the VWP of Netbeans 6.1 via the 4.3 Woodstock nbms.

2.0 Test Requirements

The Rating will be tested to meet the following requirements, standards, and tests:
  • A11Y compliant by GA
  • I18N compliant by GA
  • XHTML transitional compliant by GA
  • GUI functional test requirements
  • Design-Time test requirements
  • JSP test requirements
  • API test requirements

2.1 Tools and Methods:

The Rating will be tested using the the following tools and methods:
  • GUI tests will be performed on platform and browser configurations identified here:
    • The table of supported Browsers.
    • The table of supported Containers.
  • Functional and BUI tests will be automated using Selenium's RC tool
  • JavaScriptLint will be used to check for syntax errors in the component's javascript.
  • API tests will be written for each method in the classes of the component
  • Automated tests will be integrated into a test harness and run nightly as a regression suite
  • Automated code coverage goal of 80% (at GA) will be achieved as outlined in the project test plan

2.2 Issue Tracking

Bugs will be filed using Bugtracker (Issue Tracker) and entered against the proper Project Woodstock Revision and sub-component category.  This tool can be found on the woodstock open source site. You must have a valid user account in order to submit issues against this sub-component.

3.8 API Tests


API level unit tests will be written by QA, calling each method written for the component.  These tests include both positive and negative value arguments.  The goal is to look for consistent and proper behavior, making sure range limits are met, and correct error reporting is handed by the class. The API tests will be incorporated into the internal test harness and run nightly.   Nightly test results will be posted and available for review.  JCOV, the java code coverage tool, will be used to ensure that we meet the code coverage goales for milestones and GA in particular.
