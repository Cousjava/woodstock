1.0 Overview

This test specification describes the approach to be taken when testing the rendering of the 4.1 Project Woodstock Accordion Component (Accordion).

1.1 Component Description

The Accordion is an interface element similar to a vertical set of tabs - when you click on the header of an accordion panel, the header animates either up or down to reveal its associated container.   It generally tends to have a depth of one level unlike tabs where multilevel tabsets exist. Being rendered vertically there is space to accommodate more markup than simply the name of the tab. Thus an accordion tab can have a lot more information than a normal tab ususally does.

2.0 Test Requirements

The Accordion will be tested to meet the following requirements, standards, and tests:
  • SWAED Guidelines compliant
  • A11Y compliant by GA
  • I18N compliant by GA
  • XHTML transitional compliant by GA
  • GUI functional test requirements
  • JSP test requirements
  • API test requirements

2.1 Tools and Methods:

The Accordion will be tested using the the following tools and methods:
  • GUI tests will be performed on platform and browser configurations identified here:
    • The table of supported Browsers.
    • The table of supported Containers.
  • Functional and BUI tests will be automated using SilkTool
  • JavaScriptLint will be used to check for syntax errors in the component's javascript.
  • API tests will be written for each method in the classes of the component
  • Automated tests will be integrated into a test harness and run nightly as a regression suite
  • Automated code coverage goal of 80% (at GA) will be achieved as outlined in the project test plan

2.2 Issue Tracking

Bugs will be filed using Bugtracker (Issue Tracker) and entered against the proper Project Woodstock Revision and sub-component category.  This tool can be found on the woodstock open source site. You must have a valid user account in order to submit issues against this sub-component.

3.2 A11Y  Tests

The Accordion component needs to comply with accessibility requirements set forth in the Solaris Section 508 Readiness Assessment Form (See 1194.22 Web-based intranet and internet information and applications).

3.3  I18N Tests

Assumptions:
1. L10N has translated both the component set and example.war.  This delivery has been unjarred and resides locally on a mounted/shared  drive.
2. The user has deployed the example.war and 4.1 component jars to a host and to the container as well.

Setup Instructions:
1. deploy the example webapp to your host.
2. deploy the localized jars to the example app and edit the faces-config.xml:
     -- on your host as root:
     cd /usr/local/apps/example/WEB-INF/lib
     cp <path to unjarred i18n jar/lib/>* .

     -- edit faces-config.xml to include the supported locales:
     cd /usr/local/apps/example/WEB-INF
     vi faces-config.xml - add in supported-locale info:

     <application>
        <locale-config>
          <default-locale>en</default-locale>
          <supported-locale>de</supported-locale>
          <supported-locale>fr</supported-locale>
          <supported-locale>es</supported-locale>
          <supported-locale>it</supported-locale>
          <supported-locale>ja</supported-locale>
          <supported-locale>ko</supported-locale>
          <supported-locale>sv</supported-locale>
          <supported-locale>pt_BR</supported-locale>
          <supported-locale>zh_CN</supported-locale>
          <supported-locale>zh_HK</supported-locale>
          <supported-locale>zh_TW</supported-locale>
        </locale-config>
     </application>

3. Redeploy the example app in AppServer.
4. Set the Browser locale to one of the languages supported and start testing!

3.8 API Tests


API level unit tests will be written by QA, calling each method written for the component.  These tests include both positive and negative value arguments.  The goal is to look for consistent and proper behavior, making sure range limits are met, and correct error reporting is handed by the class. The API tests will be incorporated into the internal test harness and run nightly.   Nightly test results will be posted and available for review.  JCOV, the java code coverage tool, will be used to ensure that we meet the code coverage goales for milestones and GA in particular.
