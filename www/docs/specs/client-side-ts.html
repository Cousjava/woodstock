1.0 Overview

1.1 Client-Side Rendering Description

Client-side rendering is a process where the requests are handled dynamically and the results are shown on the fly without interacting
with the server. All the processing takes place at client-side and the page is not submitted to the server.

2.0 Test Requirements

The client-side rendering will be tested to meet the following requirements, standards, and tests:
  • SWAED Web App Guidelines compliant
  • Fully A11Y compliant by GA
  • I18N compliant by GA
  • XHTML transitional compliant by GA
  • GUI functional test requirements
  • JSP test requirements
  • API test requirements

2.1 Tools and Methods:

The client-side rendering will be tested using the the following tools and methods:
  • GUI tests will be performed on platform and browser configurations identified here:
    • The table of supported Browsers.
    • The table of supported Containers.
    • API tests will be written for each method in the classes of the component
    • Automated tests will be integrated into a test harness and run nightly as a regression suite
    • Automated code coverage goal of 80% (at GA) will be achieved as outlined in the project test plan

3.4  I18N Tests

    Assumptions:
1. L10N has translated both the component set and example.war.  This delivery has been unjarred and resides locally on a mounted/shared    drive
        2. The user has deployed the example.war and 4.1 component jars to a host and to the container as well.

Setup Instructions:
1. deploy the example webapp to your host.
2. deploy the localized jars to the example app and edit the faces-config.xml:
 -- on your host as root:
 cd /usr/local/apps/example/WEB-INF/lib
 cp <path to unjarred i18n jar/lib/>* .

 -- edit faces-config.xml to include the supported locales:
 cd /usr/local/apps/example/WEB-INF
 vi faces-config.xml - add in supported-locale info:

 <application>
 <locale-config>
 <default-locale>en</default-locale>
 <supported-locale>de</supported-locale>
 <supported-locale>fr</supported-locale>
 <supported-locale>es</supported-locale>
 <supported-locale>it</supported-locale>
 <supported-locale>ja</supported-locale>
 <supported-locale>ko</supported-locale>
 <supported-locale>sv</supported-locale>
 <supported-locale>pt_BR</supported-locale>
 <supported-locale>zh_CN</supported-locale>
 <supported-locale>zh_HK</supported-locale>
 <supported-locale>zh_TW</supported-locale>
 </locale-config>
 </application>

3. Redeploy the example app in AppServer.
4. Set the Browser locale to one of the languages supported and start testing!

   3.7 Design Time Tests
              Design time behaviour  for  components rendering client-side is not expected to change.The client-side rendering components
will  continue  to use  the  existing  html  renderers  present in runtime  and  designtime  packages  for design  time behaviour.   Hence
behaviour of  components  should not  get  affected by adding  client-side  rendering. The existing  behaviour of component  should be
tested  to see nothing breaks after adding  client-side behaviour, at  run time component should always render client-side functionality.

                Any recent build of NetBeans 6.0  will have Woodstock components incorporated.We just  need to install  latest Netbeans6.0
all the components listed in palette. Drag and drop any client-side  renderer component ,change the attributes at designtime, behaviour
for component wont change,compile and run the client-side behaviour should be seen in the browser for respective components.

   3.8 API Tests


  API level unit tests will be written by QA, calling each method written for the component.  These tests include both positive and negative
value arguments.  The goal is to look for consistent and proper behavior, making sure range limits are met, and correct error reporting is
handed by the class. The API tests will be incorporated into the internal test  harness and run nightly.   Nightly test results will  be  posted
and available for review.  JCOV, the java code coverage tool, will be used to ensure that we meet the code coverage goales for milestones
and GA in particular.
