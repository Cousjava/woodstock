Component Functional Spec - Accordion

1.0 Component Description

This document specifies the Accordion user interface component. There are two sets of interfaces in this component. A client side javascript interface that allows user to dynamically manipulate the widget and a server side JSF based component interface.

The accordion component is essentially a vertical set of tabs with a few significant differences. It generally tends to have a depth of one level unlike tabs where multilevel tabsets exist. Being rendered vertically there is space to accommodate more markup than simply the name of the tab. Thus an accordion tab can have a lot more information than a normal tab ususally does.

2.1.2 Functional requirements (required)

The number of accordaion tabs selected/open at a given point in time should be configurable. By default, only one would be open.
Accordion tab content should be able to display any information from simple HTML markup to more complex components like trees.
It should be possible to assign a title/label to each accordion tab. The contents of this label should be configurable too.
Each Accordion tab should allow for the addition of functional entities like "edit", "help", etc.
It should be possible to show or hide an accordion tab.
The width of the accordion should be configurable. The height of each accordion tab including the label height should be configurable too.
The accordion should be able to resize itself when the browser window is resized.
It should be possible to change the look of accordion tabs based on the whether they are open, close or being moused over.
When swictching from one accordion tab to another it should be possible to configure whether a trip to the server needs to be made before moving to the new tab.
It should be possible to position an accordion anywhere in a browser window.
It should be possible for accordion tabs to themselves be accordion containers (a second level accordion).
It should be possible for developers to change the contents of a given accordion tab based on certain events (clicking on a "edit" button, for example)
It should be possible to refresh the contents of the Accordion or of each individual tab.
It should be possible to reorder the accordion tabs by drag and drop.
Selecting an accordion tab should trigger events that applications can listen to.

2.1.3 Performance requirements (optional)

An accordion should perform optimally. In situations where the data for a given tab content is being loaded an appropriate message should be displayed in the content section until the data has been uploaded completely.

3.0 Component Usage

The accordion can be used for object based navigation as well as task based navigation. It is generally not recommended that accordions be used for non navigational purposes. Also, when the number of navigational tasks or objects gets too large one should consider using a tree or some other UI component as opposed to the accordion.

In the case of simple navigation the accordion tends to be on one frame (the left frame) and the objects/tasks to be navigated to on the other frame.

4.0 Client Side Widget Interface

This section outlines the client side API of the accordion and accordionTab Widget. As mentioned above, the accordion is a container widget that contains a number of accordionTab widgets. The term "accordionTab" and  "tab" and "tabContent" refer to the same entity in the rest of this document.

Accordion extends Dojo's AccordionContainer widget. AccordionTab extends Dojo's ContentPane widget. While Dojo's version of the base widgets provide most of the functionality they do not support some features that the Woodstock widgets would provide. These include:

a) Allowing multiple accordion tabs to be open/selected at the same time
b) Expanding/collapsing/refreshing the Accordion
c) Associating toolbars, edit, help functionalities with each tab.

5.3 Tag Accordion

A container for one or more accordionTab elements. The currently selected element is specified via the {@code selected} property, which may be bound to a backing bean value. This will have an impact only the first time around - subsequently the users action on the client side will override this value.

webuijsf
Tag accordion

Use the webuijsf:accordion tag to display an accordion component in the rendered HTML page. An accordion can be thought of as a vertical tab set. It can contain one of more tabs each of which can contain virtually any HTML markup. In general accordions are used for navigational purposes - each tab contains links which when clicked takes the user to different sections (tasks) of the application. In general only one accorion tab can be open at any given time. This accordion, however, allows muliple tabs to be open simultaneously. In such situations the accordion also supports "expandAll" and "collapseAll" icons which have their usual meaning. An accordion can be refreshed if the application using it so desires. Each attribute of the accordion is described in detail in the TLD doccument. It is adviseable to use an alternate navigational component if the number of tabs in the accordion are exceeding ten.

Unless otherwise specified the accordion will nto display the "expand all", "collapse all" and "refresh" icons. One has to set the respective attributes, displayRefresh, expandCollapse to true.

An accordion container. It extends the TabContainer and adds some functionality that is specific to the Accordion. An accordion can be thought of as a vertical tab set. It can contain one of more accordion tabs each of which can contain virtually any HTML markup. In general accordions are used for navigational purposes - each tab contains links which when clicked takes the user to different sections (tasks) of the application. The Accordian allows one or more tabs to be open at any given time. When the accordion is configured to allow multiple tabs to be open at any given time it supports "expandAll" and "collapseAll" icons. The expandAll icon when clicked will expand all tabs, and colapseAll, will collapse all open tabs. An accordion can be refreshed. Refreshing the accordion will cause it to render itself and all its children again. The Accordion can also be refreshed to go through all the steps of the JSF lifecycle as opposed to just the render response phase. It is adviseable to use an alternate navigational component if the number of tabs in the accordion are exceeding ten.

A container for one or more tabs, accordion tabs or menu item components. The currently selected element is specified via the selected property, which may be bound to a model value. Action listeners may be registered individually with a contained element; or, an action listener may be registered with the containing TabContainer, in which case it is notified of all tab selection actions. Each tab in the container can have arbitrary HTML elements or components or a combination of both. Typically a single tab or the entire container would go through the JSF lifecycle during an XHR request. However, in cases where a form with a contianer component is submitted the container and all its child components will go through the JSF lifecycle as usual. It is up to the application to decide what should be done in this scenario.

8.0 Component High level design (optional)

The accordion component consists of a container, referred to as the Accordion, and a set of contained elements referred to as the accordionTab. The container supports global functionalities such as:

a) rendering the outer framework
b) resizing the accordion as and when tabs get added/removed or when the window is resized
c) handle the animation as tabs are opened and closed.
d) handle drag and drop activities.
e) refresh the accordion and its contents.
f) handle expand all / collapse all.
 
Each accordionTab performs the following functions:

a) Renders itself.
b) Figures out of the content needs to be loaded from the server and if so make an XHR to load the contents.

On the server side there is a Accordion component which can have any number of AccordionTab children. The component is only allowed to have AccordionTab as children. Any other type of child is disregarded. An AccordionTab component has a label that can be specified via  facet as well. It can have any arbitrary aggregation of JSF components as its children. Needless to say, since the client side implementation has a fair amount of control the contents should ideally be widgets that have client side javascript based renderers.


Client side design:

Since accordions in general do not have too many tabs this implementation sends across each of the tab content to the client. On the client side tabs are selected or hidden by manipulating the content domNode's style (visible:hidden).

The usual Dojo based widget architecture would be applied here too. In addition there is the Dynafaces based javascript that acts as the gateway between the client and the JSF based component.

The "refresh()" function would be used whenever possible to update the contents of a tab or to present a different view of a given tab if the user selected a menu item. refresh() will be invoked on the individual tab. App developers can invoke refresh() in another tab in situations where they want the old tab's contents be saved before a new tab is selected.

Server side design:

If a form containing an accordion is submitted the container along with its children, the tabs, would go thru the JSF lifecycle just like any other component. In general the client side widget would invoke the refresh method on the container or one or more of the individual tabs in which case only those components (and their children) would go thru the lifecycle. Therein lies the real value of using Ajax in this widget.
